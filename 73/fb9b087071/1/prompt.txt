[P1] The biggest risk is your OAuth plan. I could not verify “CLI captures Google refresh token, then Convex passes it to Unipile” from Unipile’s official docs. What I did find is that Unipile documents either:
Hosted Auth / Hosted Auth Wizard, or
Google OAuth configured in the Unipile dashboard with Unipile callback URLs.
That means your current “keep existing Google OAuth and hand the token to Unipile” assumption needs proof before you commit the architecture. If it’s not supported, a lot of Phase 2 changes.
[P1] You need to store Unipile provider_id per email, not just message_id / thread_id. Unipile’s email reply docs say replies use reply_to with the provider ID of an email. Without that, “reply in same thread” is underspecified.
[P1] Don’t write a message to SQLite as sent before Convex/Unipile confirms success. Your “terminal on” flow currently writes email/enrollment/task locally before the remote send succeeds. That will create false history if the remote send fails. You want a send_intent / pending_send state first, then promote to sent on confirmation.
[P1] I would not keep mailbox_daily_usage as a true bidirectional table long-term. It’s derived state and extremely conflict-prone. After Phase 3, Convex should be authoritative for send ledger and counters; SQLite should mirror it.
[P1] Your webhook security note should probably use a shared secret header, not HMAC, unless you’ve already confirmed HMAC support elsewhere. The official Unipile webhook docs I found describe adding a secret header to authenticate webhook calls.
[P2] “SQLite and Convex are both first-class” is workable for cacheable business objects, but dangerous for send/execution state. I’d make Convex authoritative for:
sends
reply events
enrollments
tasks
suppressions
usage counters
SQLite can still be first-class for local-only/session data.
[P2] Pre-generating all follow-up drafts at enrollment time is fine for simple bumps, but it will stale fast for Nurge’s AI value prop. I’d add draft invalidation rules:
if ICP/product memory changed
if thread got new activity
if task is older than N days
if mailbox changed
[P2] Convex cron has an execution caveat: only one run of a given cron can execute at a time, and later runs may be skipped if the previous one is still running. So checkDueTasks needs to stay short and claim work in small batches.
[P3] LinkedIn is not quite “ready to flip on later without behavior changes.” Unipile’s accepted-invitation detection for LinkedIn is not real-time; their docs say it can be delayed by up to 8 hours. That affects sequence timing semantics.
What I’d Change

Make Convex the source of truth for all sequence execution tables after Phase 3. SQLite stays a mirror plus local/session store.
Add provider_id, account_id, tracking_id, and remote status fields to your synced email/send schema now.
Replace “write sent locally first” with:
local intent row
remote send
confirmation event
then mark sent
Keep sequence_drafts, but treat them as regenerable artifacts, not permanent truth.
Add one more status besides failed: pending_remote_send. You’ll need it.
Make mailbox_daily_usage derived from send events in Convex, not user-editable synced state.
Overall
The shape is strong. The two things I’d validate before building much more are:

whether Unipile really supports your planned Google token handoff flow, and
whether you’re willing to let Convex become authoritative for sequencer state.
If those two land, this is a very credible design.
