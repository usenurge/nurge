### 1. SECURITY ISSUES

**1a. Token in by_token index can be empty string -- authenticates as any logged-out user**
- File: `convex/auth.ts:69` + `convex/auth.ts:36-40`
- `clearUserToken` sets token to `""`. The `findUserByToken` query searches `by_token` index with the provided token. If an attacker sends `token: ""`, it will match any user whose token was cleared (logged out). The `validateToken` and `revokeToken` endpoints, as well as every `findUserByToken` call in `sendApi.ts`, `connect.ts`, and `enrich.ts`, are all vulnerable. A logged-out user's record becomes matchable by an empty-string token.
- **Fix:** `findUserByToken` should reject empty/blank tokens before querying, or `clearUserToken` should set a non-guessable sentinel value (or delete the field).

**1b. Gmail Pub/Sub webhook has no authentication**
- File: `convex/gmailWebhook.ts:14-47`
- The `gmailPubSubWebhook` handler performs zero authentication. Anyone can POST a crafted payload to `/api/gmail-webhook` to mark any email address as needing sync. Google Pub/Sub push subscriptions support JWT verification via the `Authorization` header, but it is not checked here.

**1c. Unipile webhook secret check is optional (bypassed when env var is missing)**
- File: `convex/webhook.ts:8-14`
- `if (secret && authHeader !== secret)` -- if `UNIPILE_WEBHOOK_SECRET` is not set, the check is skipped entirely. Any external caller can invoke the webhook. This should probably reject requests when the secret is not configured rather than accepting all.

**1d. Unipile account callback has no authentication**
- File: `convex/webhook.ts:107-148`
- `unipileAccountCallback` is a public HTTP endpoint that accepts any POST with an `account_id` and `name` field. A malicious caller can register arbitrary Unipile account IDs against any userId by guessing or brute-forcing the userId (which is a Convex document ID embedded in the `name` field from `requestConnectionLink`). There is no signature verification or shared secret.

**1e. `enrollments.ts` mutations are public mutations, not internal**

### 2. RACE CONDITIONS

**2a. Claim-then-send race in dispatcher (TOCTOU)**
- File: `convex/dispatcher.ts:85-107`
- The cron `checkDueTasks` is an `internalAction` that runs multiple separate query+mutation steps. Convex actions are not transactional across multiple runQuery/runMutation calls.

**2b. syncVersion increment race across all files**
- Files: `convex/helpers.ts:19-37`, `convex/send.ts:125-196`, `convex/sendApi.ts:472-496`, `convex/webhook.ts:300-370`, `convex/connect.ts:278-305`, many others.

But in `registerUnipileAccount` (webhook.ts:374-533), the function reads syncState at line 436-440, writes version, and then reads syncState **again** at line 484-488 (shadowing the outer `version` and `syncState` const). The second read sees the DB state as of the mutation start (snapshot isolation within the mutation), not the write from line 459-462. This means the second `version` at line 488 will be stale -- it re-reads the same old value, potentially producing duplicate sync versions.

**2c. Non-atomic check-then-act in `processReply`**
- File: `convex/webhook.ts:164-273`

### 3. LOGIC ERRORS

**3a. `generateToken` modulo bias**
- File: `convex/auth.ts:14-19`
- `chars[b % chars.length]` where `chars.length = 62`. Since `256 % 62 = 8`, the first 8 characters (a-h) have a slightly higher probability than the remaining 54. For a security token, this is a minor but real bias. Using rejection sampling or `crypto.randomUUID` would be better.

**3b. `listEmailsAfterVersion` does NOT filter by userId in the index**
- File: `convex/connect.ts:225-242`
- The query uses `.withIndex("by_sync_version")` which does not include userId, then does a post-hoc `.filter()` in JavaScript. This means it scans ALL emails in the system by sync version, loads them all into memory, and then filters by userId. For a multi-tenant system this is both a performance problem and a data leak risk (the action could time out, or if the filter logic had a bug, other users' emails could leak).

**3c. `listEmailsAfterVersion` userId type mismatch**
- File: `convex/connect.ts:226-228`
- The arg `userId` is `v.string()`, but the calling code at line 152 passes `user._id` which is a Convex `Id<"users">`. In the filter at line 238, it compares `e.userId === args.userId`. Whether this works depends on whether `user._id` serializes to the same string stored in `e.userId`. If `emails.userId` stores a plain string (like from auth), but `user._id` is a Convex ID object, the comparison could fail silently, returning zero emails.

**3d. `countSendsForDay` uses compound index incorrectly**
- File: `convex/helpers.ts:80-88`
- The `by_mailbox_day` index is defined as `["mailboxId", "sentAt"]` (schema.ts:88). The query uses `.eq("mailboxId", ...).gte("sentAt", ...).lte("sentAt", ...)`. However, Convex index range queries require `.eq()` on all prefix fields followed by a single range operation. Using both `.gte()` and `.lte()` on the same field within `withIndex` may not work as expected in Convex -- you need to chain them: `.eq("mailboxId", x).gte("sentAt", start)` and then apply `.lte()` as a post-filter or use `.filter()`. This could silently return incorrect counts.

**3e. `getWorkingLinkedInAccount` ignores the Convex-Unipile cross-check it claims to do**
- File: `convex/enrich.ts:33-57`
- The function fetches all LinkedIn accounts from Unipile API, but then at line 48-49 does `for (const id of linkedinAccounts) { return id; }` -- it returns the FIRST active Unipile account regardless of whether it matches any account in `ids` (the Convex accounts for this user). The comment says "Return first account that's both in Convex and active in Unipile" but the code never checks if `id` is in `ids`. A user could end up using another user's LinkedIn account.

**3f. `recordSendSuccess` does not advance to the NEXT step**
- File: `convex/send.ts:164`
- `const newStep = task.stepNumber;` sets the enrollment's `currentStep` to the just-completed task's step. But it doesn't schedule/create the next task or update `nextDueAt` for the next due task. The enrollment's `nextDueAt` remains unchanged (line 172). Future tasks exist but `nextDueAt` will be stale. Same issue in `sendApi.ts:698`.

**3g. `processReply` matches by contactEmail only, not by thread**
- File: `convex/webhook.ts:176-180`
- The `threadId` and `providerId` are passed as args but never used for matching enrollments. An email from the same contact on a completely unrelated thread will stop ALL active enrollments for that contact across all sequences, even if the reply is to a personal thread unrelated to the sequence.

The backfill `UPDATE` statements (e.g., `UPDATE mailboxes SET warmup_started_at = created_at WHERE warmup_started_at = ''`) run on every `Open()` call, not just when the column is first added. **45. DROP INDEX + CREATE UNIQUE INDEX is not atomic with respect to concurrent readers (lines 138-143)**
```go

```
While this runs in a transaction, if there are duplicate `(atom_type, scope)` rows in `knowledge_atoms` (from before the unique constraint), the `CREATE UNIQUE INDEX` will fail, rolling back the entire migration transaction and preventing the database from opening. **46. No migration version tracking (entire file)**
The migration system uses column-existence checks (`hasColumnTx`) rather than a migration version table. This means:
- Migrations are idempotent but checked individually, making startup slower as the migration list grows.
- There's no way to run one-time migration code (e.g., data transforms) that shouldn't re-run.
- If a migration is removed from the list, its column still exists but can't be tracked.
