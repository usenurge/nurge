### 1. COMPLETENESS -- Significant Gaps

**Critical: No Unipile API integration details**

The spec says Unipile LinkedIn search and profile fetch were "tested and return real data" and that `nurge connect linkedin` is built and working. However, the spec never specifies:

- How to actually call the Unipile API from Go code (no endpoint patterns, no auth header format, no SDK reference)
- Where the Unipile API token/credentials live (the connect flow goes through Convex, but enrichment needs to call Unipile directly from the Go binary)
- The actual JSON response shapes from Unipile's people search and profile endpoints
- Rate limiting behavior beyond the stated "1,000 profiles/day" number

This is the single biggest gap. The entire spec hinges on "Unipile LinkedIn" as Tier 1 enrichment, but a developer would have to reverse-engineer the API integration from scratch.

**Critical: No definition of "nurge knowledge seed" command behavior**

The spec references `nurge knowledge seed` (line 181) as the CLI command for storing transformation models and signal playbooks. The existing `cmd/knowledge.go` and `internal/assembly/knowledge.go` support 8 atom types: `tone`, `hooks`, `positioning`, `personas`, `objections`, `timing`, `compliance`, `structure`. The spec introduces two new knowledge atom types -- "transformation models" (line 205) and "signal playbook" (line 206) -- but never defines:

- What atom type string they use
- What their content JSON schema looks like
- Whether they extend `ValidAtomTypes` in `internal/assembly/knowledge.go` or use a different mechanism
- How signal playbook atoms relate to the signal definitions in S3 (lines 87-120)

**Important: The enrichment waterfall is disconnected from Unipile**

The existing waterfall in `/Users/rejoesudarsan/Projects/nurgesh/internal/enrich/waterfall.go` references Apify-based providers (`apify-linkedin-person`, `apify-linkedin-company`, `apollo-email`). The spec says to use Unipile LinkedIn as Tier 1 enrichment, but does not specify:

- Whether Unipile replaces the Apify providers in the waterfall or adds alongside them
- What the new provider YAML definitions look like for Unipile LinkedIn search/profile
- Whether WebSearch and WebFetch (Tiers 2 and 3) are even Go-binary-callable or are Claude tool calls

This last point is fundamental. WebSearch and WebFetch are Claude Code tool calls, not Go functions. The enrichment waterfall in `internal/enrich/enrich.go` is a Go binary executing providers sequentially. Steps 4-6 of the enrichment waterfall (lines 161-174) describe WebSearch and WebFetch operations, but these cannot be called from Go code -- they are Claude capabilities. The spec conflates two different execution contexts.

**Important: S6 "Generate thin skill wrappers + seed knowledge atoms" is entirely undefined**

Line 133 lists "S6: Generate thin skill wrappers + seed knowledge atoms" as a setup step but provides zero detail on what this step actually does. ### 2. CONSISTENCY -- Two Contradictions and One Tension

**Contradiction: Two specs for the same date, overlapping scope**

There are two specs dated 2026-03-26:
- `2026-03-26-context-assembly-architecture-design.md` -- the detailed architecture spec with code-level Go types, DB schema, thin skill templates, resolve pipeline, enrichment engine, tiering engine
- `2026-03-26-enrichment-context-assembly-design.md` (this spec) -- overlaps significantly but introduces new concepts (intelligence build, signals, transformation models)

These specs overlap on: context assembly, enrichment, knowledge atoms, and setup changes. But they diverge on details. The architecture spec's waterfall uses Apify/Apollo providers. This spec uses Unipile. Which is canonical? A developer reading both specs would be confused about which to follow.

**Contradiction: CLI-as-API principle vs. the enrichment waterfall**

CLAUDE.md states: "Go binary = dumb (sync, store, serve). Claude = smart (classify, converse, generate)." The spec's enrichment waterfall (lines 152-174) puts Claude reasoning (Step 6) as part of a `nurge context assemble` call. But the Go binary cannot invoke Claude reasoning. Similarly, WebSearch and WebFetch in steps 4-5 are Claude tools, not Go library calls. The waterfall diagram implies a single orchestrated pipeline, but in practice it would need to be split: Go binary handles Steps 1-3 (Unipile API calls), and Claude handles Steps 4-6 (web search, page fetch, reasoning). This split is not acknowledged or designed for.

**Tension: Setup redesign scope vs. existing Phase B**

CLAUDE.md Phase B (lines 47-59) defines an 11-step setup with specific ordering. This spec's "Intelligence Build" (lines 60-133) replaces steps 6-10 with S1-S6, fundamentally changing the setup flow. The spec does not mention which existing Phase B steps survive, which are replaced, or how the transition works for users who already completed the old setup.

### 3. CLARITY -- Implementation Ambiguities

**The "Unipile LinkedIn provider" (line 198) has no implementation guidance**

Item 1 in "What Needs to Be Built" says to create a Unipile LinkedIn provider for the enrichment waterfall. But the existing provider system (`internal/enrich/provider.go`) is YAML-based and designed for HTTP API providers with simple input/output mappings. Unipile LinkedIn search requires multiple sequential API calls (search for person, then fetch full profile, then search for company employees, then search for former employees). This does not fit the current single-call-per-provider model at all.

A developer would need to either: (a) extend the provider abstraction to support multi-step providers, or (b) build Unipile integration as a special case outside the provider registry. The spec does not indicate which approach, or even acknowledge the mismatch.

**Signal detection logic (line 203) is described narratively but not structurally**

The signal definitions in S3 (lines 87-120) show example YAML for signals with `sources`, `recency`, and `intrigue_angle`. But there is no specification of:
- Where signal definitions are stored (knowledge_atoms? A new table? YAML files on disk?)
- What the signal detection algorithm looks like in Go code
- How "recency: must be within last 6 months" is evaluated against enrichment data
- What the output of signal detection looks like (the data structure a skill consumes)

**The `nurge enrich <prospect>` and `nurge research <company>` commands (line 199) are mentioned but not specified**

These are new CLI commands. The existing `nurge enrich` is not registered in CLAUDE.md's CLI command list. Their flags, output format, and relationship to the existing `nurge context assemble` command are undefined.

### 4. SCOPE -- Appropriate With One Risk

The spec is appropriately scoped for a design document. It covers enrichment (Unipile), setup redesign (intelligence build), and runtime assembly. These are tightly coupled and belong together.

**Risk: The setup redesign (S1-S6) is enormous**

Steps S1 through S6 describe a fundamentally new setup experience where Claude does research, proposes ICP, builds personas with transformation models, configures signal playbooks, runs a live test, and then seeds knowledge and generates skills. Each of these steps is a multi-turn conversation with real API calls. This is probably 2-3 weeks of work for the setup flow alone. The spec treats it as one section, which could lead to a monolithic implementation attempt rather than incremental delivery.

I would recommend splitting S1-S6 into separate implementation tasks with clear "what's minimally shippable" criteria for each.
