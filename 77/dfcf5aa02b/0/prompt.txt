⏺ Here's how OpenClaw solves it — three layers:                                                      
                                                                                                     
  OpenClaw's Connector Architecture                                                                  
                                                                                                     
  Layer 1: Declarative manifest (openclaw.plugin.json)                                               
  - Declares ID, auth methods, required env vars, config schema                                      
  - Lightweight — can be read without loading any code                                               
  - Enough for onboarding wizards to show "what can you connect?"                                    
                                                                                                     
  Layer 2: Package metadata (package.json)                                                           
  - Discovery info, entry points, UI labels/icons                                                    
  - Drives directory-based auto-discovery — drop a folder in extensions/, it's found automatically   
                                                          
  Layer 3: TypeScript code (index.ts)                                                                
  - Calls definePluginEntry() with a register(api) callback
  - Imperatively registers capabilities: api.registerProvider(), api.registerTool(),                 
  api.registerChannel()                                                             
  - This is where OAuth flows, API calls, and data logic actually live                               
                                                                      
  Key patterns:                                                                                      
  - Directory-based discovery — scan for manifests, zero core code changes to add a connector        
  - Shared auth helpers — createProviderApiKeyAuthMethod(), createVpsAwareOAuthHandlers() — so each  
  connector doesn't reinvent OAuth                                                                   
  - Skills reference tools, not connectors directly — skills are SKILL.md docs that say "use the     
  slack tool", the plugin registered that tool            
                                                                                                     
  The tension for Nurge                                   
                                                                                                     
  OpenClaw connectors are still code (TypeScript), not pure YAML. The manifest is declarative, but   
  the actual sync/auth logic is imperative. They just made the discovery and metadata declarative so
  the system can reason about connectors without loading them.                                       
                                                          
  For Nurge, pure YAML could work for auth (OAuth endpoints, scopes, API key prompts) but the sync   
  logic (Gmail pagination, HubSpot field mapping, incremental cursors) needs Go code.
                                                                                                     
  Does this change your thinking on the YAML-only approach, or are you imagining connectors that only
   need auth (no complex sync)?

⏺ I think the tiered approach is the right call. Here's my reasoning:                                
                                                          
  Tier 1 (Go binary) — only Gmail/Outlook. These are the only connectors that need background sync,  
  complex pagination, incremental cursors, and local DB storage. There's maybe 2-3 of these ever.
  Worth the Go code.                                                                                 
                                                                                                     
  Tier 2 & 3 (Claude-native) — everything else. CRMs, enrichment tools, sequencing platforms — Claude
   already knows how to call REST APIs. A YAML file defines the auth, a SKILL.md teaches Claude how
  to use the API. No Go code, no binary release, no compile step.                                    
                                                          
  Think about it from the user's perspective: "I use Pipedrive" → Claude runs nurge connect pipedrive
   (generic OAuth engine reads the YAML) → token stored → Claude reads the Pipedrive skill → knows
  how to push/pull contacts, log activities, update deals. All at conversation time, not sync time.  
                                                          
  Why this works for outbound specifically:                                                          
  - CRM reads/writes happen in conversation context — "log this email to the deal" or "pull the
  contact list for this account." Claude is already there.                                           
  - Enrichment is always on-demand — "research this prospect" → call Apollo API → return data. No
  sync needed.                                                                                       
  - The only thing that truly needs background sync is the email mailbox itself.                     
                                    
  The YAML connector definition could look like:                                                     
  id: pipedrive                                           
  name: Pipedrive                                                                                    
  type: crm                                                                                          
  auth:                             
    method: oauth2                                                                                   
    authorize_url: https://oauth.pipedrive.com/oauth/authorize
    token_url: https://oauth.pipedrive.com/oauth/token        
    scopes: [contacts:read, deals:read, activities:write]                                            
                                                         
  Go binary handles: generic OAuth flow, token storage, token refresh. That's it. Claude handles     
  everything else through skills.                                                                    
                                                                                                     
  The big win: You (or Claude) can ship a new connector in minutes — one YAML file + one SKILL.md. No
   release cycle.
